## Approach

Rewrote the bootstrap inner loop (`src/algorithm/fft.rs` + `mod.rs`); **params unchanged**
from entry 0001 (still 128-bit, σ-margin ~4.6 bits). Benchmarking [rs-tfhe](https://github.com/thedonutfactory/rs-tfhe)
on the same machine was the key: it hits **14.8 ms / gate with rustfft** (its SPQLIOS AVX path
is commented out), and its per-FFT cost (~1.9 µs) is close to ours — so the gap was **not the
FFT** but the per-CMux glue. Fixes, in order of impact:

1. **Inline, batched gadget decomposition** — was 2048 `decompose()` calls (each returning an
   `[i64;8]`) per CMux; now one inline pass with sequential writes. (Biggest win.)
2. **Monomial rotation via `copy_from_slice`** — two contiguous regions (vectorized memcpy +
   a negation loop) instead of branchy scattered writes.
3. **Split-format (SoA) spectra** end-to-end (`[re(M)|im(M)]`) so the GGSW external-product
   pointwise MAC auto-vectorizes (matches rs-tfhe's `fma_in_fd`).
4. **Hand-written radix-4 SIMD FFT** (NEON / AVX2+FMA / scalar, runtime-dispatched; fused
   negacyclic twist/untangle; bit-reversal folded into the twist-pack load). Naive-DFT unit
   test pins correctness.

Result on this arm64 (M3): **32.4 → ~22 ms** (~32% faster). Remaining gap to rs-tfhe's 14.8 ms
is structural: our n=768 (vs 700, +10%) and 4-message space, the q=2^64 float→torus reduce
(vs their q=2^32), and `Vec<Vec<u64>>` vs fixed `[T;N]` poly layout. KS+extract is ~1.85 ms
(comparable to theirs). x86 uses the 4-wide AVX2 FFT path (validated via cross-compile; measured on CI).
